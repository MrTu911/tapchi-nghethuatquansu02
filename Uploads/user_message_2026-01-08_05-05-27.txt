# 🧾 **BÁO CÁO ĐÁNH GIÁ & LỖI CẦN CẢI THIỆN (01/2026)**

---

## 🧱 I. KIẾN TRÚC TỔNG THỂ (ARCHITECTURE)

### ✅ Điểm mạnh

* Hệ thống **Next.js + Prisma + PostgreSQL** thiết kế đúng hướng (SSG + SSR + API routes).
* Prisma schema rất chi tiết, mô hình hóa đầy đủ quy trình xuất bản khoa học.
* Module chia theo workflow (Author / Reviewer / Editor / CMS / Repository / Security).

### ⚠️ Các điểm cần cải thiện

| Lỗi / Thiếu sót                                | Mức độ        | Mô tả & Hướng xử lý                                                                                                    |
| ---------------------------------------------- | ------------- | ---------------------------------------------------------------------------------------------------------------------- |
| **1. Prisma schema quá khổng lồ (7000+ dòng)** | ⚠️ Trung bình | Cần chia nhỏ thành **modules**: `schema.workflow.prisma`, `schema.cms.prisma`, `schema.security.prisma` để dễ bảo trì. |
| **2. Không có migration tự động test**         | ⚠️ Cao        | Cần chạy `npx prisma migrate deploy` trong CI/CD, tạo test DB (Docker Compose).                                        |
| **3. Chưa có caching layer**                   | ⚠️ Trung bình | Dữ liệu như bài viết, reviewer, dashboard nên có `react-query` hoặc `SWR` caching phía client.                         |
| **4. API route phân tán**                      | ⚠️ Trung bình | Nên gom nhóm theo domain: `/api/editor/*`, `/api/reviewer/*`, `/api/repository/*`.                                     |
| **5. Chưa có layer service riêng**             | ⚠️ Trung bình | Tách logic khỏi route (hiện nhiều Prisma query viết trực tiếp trong `page.tsx`).                                       |
| **6. Thiếu hệ thống logs thống nhất**          | ⚠️ Cao        | Dù có `AuditLog` model, nhưng chưa có `logger` ở backend (`pino` hoặc `winston`).                                      |
| **7. Thiếu queue xử lý nền (background jobs)** | ⚠️ Cao        | Email reminder, sync repository, plagiarism check nên chạy async (BullMQ hoặc cron job).                               |

## 🧩 II. MODULE WORKFLOW (SUBMISSION + REVIEW + EDITOR)

### ✅ Điểm mạnh

* Giao diện **Editor Dashboard** hoàn chỉnh (Kanban Board, Stats, Urgent Items).
* Reviewer Management có phân tích hiệu suất, phản hồi trung bình.
* Workflow phù hợp mô hình OJS (Editor → Reviewer → Decision → Production).

### ⚠️ Cần cải thiện

| Vấn đề                                             | Mức độ        | Mô tả / Đề xuất                                                                                                        |
| -------------------------------------------------- | ------------- | ---------------------------------------------------------------------------------------------------------------------- |
| **1. Logic workflow chưa tách riêng service**      | ⚠️ Cao        | Nên có `WorkflowService` quản lý state machine (NEW → UNDER_REVIEW → REVISION → ACCEPTED → IN_PRODUCTION → PUBLISHED). |
| **2. Thiếu bảng `SubmissionTransitionLog`**        | ⚠️ Trung bình | Lưu lịch sử thay đổi trạng thái với người thao tác, thời gian.                                                         |
| **3. Không có cơ chế tự động deadline reminder**   | ⚠️ Cao        | Nên có cron job gửi email khi sắp hết hạn phản biện (`Deadline` model có sẵn).                                         |
| **4. Không có UI quản lý vòng phản biện (rounds)** | ⚠️ Trung bình | Mỗi vòng nên hiển thị riêng (Review Round 1 / 2 / 3).                                                                  |
| **5. Giao diện ra quyết định (Editor Decision)**   | ⚠️ Trung bình | Cần form riêng `/editor/decision/[submissionId]` để chọn Accept/Minor/Major/Reject.                                    |
| **6. Cần audit trail cho mọi thay đổi**            | ⚠️ Cao        | Khi Editor thay đổi trạng thái hoặc Reviewer nộp phản hồi → log lại `AuditLog`.                                        |

## 📚 III. MODULE REPOSITORY (Kho dữ liệu bài báo)

### ✅ Điểm mạnh

* Kiến trúc “Open Repository with Controlled Download” rất chuẩn học thuật.
* CSDL thiết kế đúng (`ArticleRepository`, `RepositoryFile`).
* Có logic phân quyền tải (session-based).

### ⚠️ Cần cải thiện

| Vấn đề                                             | Mức độ        | Đề xuất                                                                   |
| -------------------------------------------------- | ------------- | ------------------------------------------------------------------------- |
| **1. Chưa có giao diện thống kê**                  | ⚠️ Trung bình | Tạo `/dashboard/repository/stats` hiển thị biểu đồ: lượt xem / tải / năm. |
| **2. Cần tách PDF khỏi public folder**             | ⚠️ Cao        | Nên lưu file trong `storage/` hoặc S3, tải qua API bảo mật.               |
| **3. Chưa có tính năng import bài cũ**             | ⚠️ Trung bình | Viết script import từ Excel hoặc BibTeX.                                  |
| **4. Thiếu SEO metadata cho mỗi bài**              | ⚠️ Trung bình | Thêm `<meta name="description">`, `<meta property="og:title">` cho SSR.   |
| **5. Chưa có gợi ý bài tương tự (related papers)** | ⚠️ Thấp       | Dùng TF-IDF hoặc cosine keywords matching.                                |

## 🔒 IV. MODULE BẢO MẬT (SECURITY & COMPLIANCE)

### ✅ Điểm mạnh

* Có `SecurityAlert`, `TwoFactorAuth`, `ApiToken`, `RoleEscalationRequest` — rất đầy đủ.
* Phân vai rõ ràng (Role enum đầy đủ).

### ⚠️ Cần cải thiện

| Lỗi / Thiếu sót                              | Mức độ        | Hướng xử lý                                                  |
| -------------------------------------------- | ------------- | ------------------------------------------------------------ |
| **1. Chưa triển khai 2FA thực tế**           | ⚠️ Cao        | Dù có model, nhưng chưa có giao diện (OTP / TOTP).           |
| **2. ApiToken chưa có middleware bảo vệ**    | ⚠️ Cao        | Cần middleware xác thực Bearer token ở `/api/*`.             |
| **3. SecurityAlert chưa ghi tự động**        | ⚠️ Trung bình | Khi đăng nhập sai >5 lần, nên tự động log BRUTE_FORCE.       |
| **4. Chưa có Content Security Policy (CSP)** | ⚠️ Cao        | Thêm headers bảo mật (Next middleware).                      |
| **5. Chưa có kiểm tra quyền RBAC runtime**   | ⚠️ Cao        | Viết `checkPermission(user, "submissions.edit")` middleware. |

| **5. Nhiều page SSR nặng (fetch trực tiếp Prisma)**      | ⚠️ Trung bình | Tách sang API route để load dữ liệu client-side. |

## 🧩 VII. KIỂM TRA ĐẠO VĂN (PLAGIARISM)

### ✅ Hiện trạng:

* Có bảng `PlagiarismReport` & `PlagiarismCheck` trong schema.
* Nhưng chưa có code logic, API hay UI nào.

### ⚠️ Cần triển khai 3 lớp:

| Giai đoạn      | Mục tiêu                      | Gợi ý công nghệ             |
| -------------- | ----------------------------- | --------------------------- |
| 1️⃣ Extraction | Trích xuất text từ PDF        | `pdf-parse`, `textract`     |
| 2️⃣ Comparison | So sánh với ArticleRepository | Cosine Similarity / TF-IDF  |
| 3️⃣ Report     | Tạo PDF hoặc JSON báo cáo     | `puppeteer` hoặc `html-pdf` |
